# 项目需求文档

## 1. 源码审查
- 阅读flow源码并报告
- 再次检查自动构建(轮询)部分的代码是否有bug

## 2. 组织功能
给项目列表加一个字段，叫做组织ID，新建一个组织表，创建项目的时候，可以选择已有组织和新组织，新组织需要手动输入，项目提交后，如果有新组织，会把这个组织插进组织表（新增一个ID），如果表里组织名已经存在，报错已存在xx组织请选择已有组织。

## 3. 项目列表TAB筛选
项目列表中，测试环境和线上环境为一级TAB，添加组织为二级TAB筛选。

## 4. 组织必填
新增项目时组织为必填项。

## 5. 用户权限树形结构
用户的权限设置改为树形结构，组织-项目的形式展示，勾选组织则自动勾选组织下所有项目，但是数据库user.json的projectPermissions始终只保存项目id。

## 6. 自动清理孤立组织
每次创建/编辑/删除项目的时候，操作完之后，检查一遍organizations.json是否存在没有被任何项目引用的组织，如果有直接删除那些组织。

## 7. UI美化
美化项目列表的UI，让测试环境正式环境两个按钮的border跟下面的内容合为一体，像那种卡片切换，切换到对应tab，下border就消失，使tab跟下面的内容融为一体一样。按钮可以方正一点，圆角小一点。

## 8. 数据目录
确认删除data目录后项目是否还能正常运行（答案：可以，会自动创建默认数据）。

## 9. 公开项目前检查敏感信息
检查项目中是否有敏感信息（密码、API Key、密钥等），确保可以安全公开。

## 10. 编写 README
编写详细的 README.md 文档，包含功能介绍、安装部署、API 文档、使用指南等。并在文档中提醒用户首次登录后修改默认密码。

## 11. Windows 部署兼容
项目需要支持部署在 Windows 电脑上，脚本执行需要兼容 Windows 环境。

### 遇到的问题
1. 脚本使用了 Linux 的 `set -e` 命令，Windows cmd 不支持
2. PowerShell 旧版本不支持 `&&` 命令连接符
3. nvm-windows 需要在终端（cmd/powershell）中执行
4. `where pnpm >nul 2>&1` 重定向语法在 cmd 中有兼容性问题，导致后续命令不执行

### 最终方案
- Linux/Mac：使用 bash + nvm 管理 Node.js 版本
- Windows：直接执行用户脚本，不做额外的环境检查和处理，由用户保证环境正确

## 12. 跨平台执行方案调研
- 尝试使用 `execa` 包解决跨平台问题，但 API 变化大，兼容性有问题
- 最终使用 `child_process.exec`，与之前成功的方案一致
